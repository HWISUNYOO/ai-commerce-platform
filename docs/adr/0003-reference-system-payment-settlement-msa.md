# 3. 레퍼런스 시스템은 결제·정산 MSA로 한다

- 상태: 채택(Accepted)
- 날짜: 2026-06-28

## 배경

오케스트레이터(Agent)의 능력을 증명하려면, Agent가 실제로 설계·구현·배포·운영하는
구체적인 타겟 시스템이 필요하다. 도메인 선택지로 결제·정산(전문 분야)과
일반 커머스(면접관 친숙) 등이 있었다.

## 결정

**결제·정산 MSA**를 레퍼런스 시스템으로 한다. 이유:

1. 프로젝트 오너의 실무 전문성(신세계면세점 매출/정산, 카드 승인·매입, VAN 연계,
   대용량 트랜잭션)을 그대로 활용 — 도메인 깊이가 차별점이 된다.
2. 결제·정산은 멱등성, 정합성, 이벤트 기반 처리, 대사(reconciliation) 등
   백엔드 난이도가 높은 주제를 자연스럽게 포함한다.

## 잠정 서비스 분해(초안, 추후 ADR로 확정)

- `payment-service` — 결제 승인/취소, 멱등성 보장
- `ledger-service` — 원장/전표 기록
- `settlement-service` — 정산/대사 배치
- `notification-service` — 이벤트 기반 알림
- 공통: Kafka(이벤트), Redis(멱등키·캐시), PostgreSQL(원장),
  Elasticsearch/OpenSearch(거래 검색)

## 결과

- 도메인 전문성 + 모던 스택을 한 시스템에서 동시 증명.
- 결제 도메인의 정확성 요구가 높아 구현 난이도가 올라간다 — 의도된 트레이드오프.
