# 8. 로컬 인프라 포트는 루프백(127.0.0.1)에만 바인딩한다

- 상태: 채택(Accepted)
- 날짜: 2026-07-05

## 배경

로컬 개발용 docker-compose 에서 PostgreSQL/Redis/Kafka 포트를 `"5432:5432"` 형태로
게시했다. Docker 의 이 표기는 **`0.0.0.0`(모든 네트워크 인터페이스)** 에 바인딩되어,
호스트가 속한 네트워크(사내망·공유망, 경우에 따라 인터넷)에서 접근 가능한 상태가 된다.

개발 중 이 상태에서 **기본 자격증명(postgres/postgres)** 을 쓰던 PostgreSQL 이
자동 스캔 봇에 의해 침해되었다. 봇은 모든 데이터베이스를 삭제하고 `readme_to_recover`
DB에 비트코인 몸값을 요구하는 노트를 남겼다(전형적인 노출 DB 랜섬웨어 패턴 — 실제로는
백업 없이 삭제만 한다). 다행히 테스트 데이터였고 Flyway 로 스키마가 재생성되어 실피해는
없었다.

## 결정

로컬 docker-compose 의 **모든 호스트 포트 게시를 `127.0.0.1:PORT:PORT` 로 제한**한다.
이렇게 하면 해당 포트는 **로컬 머신에서만** 접근 가능하고 외부 네트워크에는 노출되지 않는다.

- 적용 대상: postgres(5432), redis(6379), kafka(9092), kafka-ui(8090), 서비스(8081~8085).
- 컨테이너 간 통신은 도커 네트워크에서 **서비스명**(`postgres`/`redis`/`kafka`)으로
  이루어지므로, 호스트 포트를 로컬로 제한해도 서비스 동작에는 영향이 없다.

## 근거

1. **최소 노출 원칙** — 개발용 데이터스토어를 외부에 열 이유가 없다.
2. 특히 **인증이 없거나 약한 서비스**(기본 비번 Postgres, 인증 없는 Redis)는 노출 시
   즉시 표적이 된다.
3. 실제 침해를 겪었고, 근본 원인이 "0.0.0.0 노출 + 기본 자격증명"이었다.

## 결과 / 후속

- 로컬에서는 이 조치로 충분하다(외부 도달 불가).
- **운영/공유 환경**에서는 추가로: 강한 자격증명(비밀은 커밋 금지, `.env`/시크릿 매니저),
  네트워크 정책, Redis AUTH, Kafka 인증 등을 적용한다(Phase 5 AKS 단계에서 반영).
- 교훈: "로컬이니까 괜찮다"는 없다. 포트 게시 표기 하나가 노출 여부를 가른다.
