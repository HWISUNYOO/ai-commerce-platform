# 2. 서비스 구현 스택을 Spring Boot 3 + Java 21로 한다

- 상태: **일부 대체(Superseded in part)** — SDLC 오케스트레이터 콘셉트는 ADR-0007로 폐기. 스택 결정은 유효.
- 날짜: 2026-06-28 (개정: 2026-06-29)

> **개정 노트 (2026-06-29):** 본 ADR은 원래 "SDLC 오케스트레이터 Agent"를 Spring Boot로
> 구현한다는 결정이었다. 프로젝트가 **일반 상품 커머스 MSA**(ADR-0003)로 전환되면서
> 오케스트레이터 콘셉트는 폐기되었다(ADR-0007). 다만 아래 **"모든 백엔드 서비스를
> Spring Boot 3.5.x + Java 21로 구현한다"**는 핵심 스택 결정은 그대로 유효하며, 전 서비스에
> 적용된다. LLM 연동 방식은 ADR-0005(LLM 백엔드 추상화)로 대체되었다.

## 결정 (유효)

모든 마이크로서비스를 **Spring Boot 3.5.16 + Java 21**로 구현한다.

근거:

1. 타겟 이직 시장의 핵심 스택이 Java 17/21 + Spring Boot 3.x다.
2. LLM 호출은 Spring의 `RestClient`로 직접 구현하여 의존성을 최소화한다 (ai-service).
3. Initializr 기본값이 Boot 4.1.0이지만, 시장 현실(3.x 압도적)을 고려해 3.5.x로 핀한다.

## 결과

- 전 서비스가 동일한 런타임/빌드(Java 21 + Gradle + Spring Boot 3.5.x)를 공유해 일관성 확보.
- Boot 4가 아닌 3.x 선택은 "왜 그랬나"를 면접에서 설명할 수 있는 포인트가 된다.
