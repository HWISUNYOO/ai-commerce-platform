# 성능 테스트 결과

[k6](../infra/k6/)로 API에 부하를 걸어 측정한 결과. 대상은 로컬 쿠버네티스 배포
(Ingress `http://localhost`), 관측은 Prometheus + Grafana.

> 로컬 단일 노드 환경의 수치이므로 절대값보다 **경향·정합성** 관찰이 목적이다.

## 1. 상품 조회 (읽기 · Redis 캐시)

`infra/k6/browse.js` — 50 VU까지 램프업하며 상품 목록/상세 조회 (2분).

| 지표 | 값 |
|------|-----|
| 총 요청 | 9,014 |
| 처리량 | 74.9 req/s |
| 에러율 | 0% |
| 지연 avg / p95 | 1.78ms / **2.47ms** |
| 임계값 | p95 < 500ms ✅ / 에러율 < 1% ✅ |

상품 조회는 `@Cacheable`(Redis)로 캐싱되어 p95 2.47ms로 매우 빠르다.

## 2. 주문 생성 (쓰기 · Saga)

`infra/k6/checkout.js` — 초당 5건 주문 유입 (1분, 총 300건). 대상 상품 재고는 49.

| 지표 | 값 |
|------|-----|
| 총 주문 접수 | 300 |
| HTTP 실패율 | 0% |
| 주문 접수 지연 avg / p95 | 7.48ms / **9.89ms** |
| 임계값 | p95 < 800ms ✅ / 실패율 < 5% ✅ |

주문 접수(동기 응답)는 빠르고, 재고예약→결제→적립은 Kafka로 **비동기** 처리된다.

### 부하 중 Saga 정합성 (핵심 관찰)

재고(49)를 초과하는 300건이 몰렸을 때 Saga의 최종 처리 결과:

| 결과 | 건수 | 설명 |
|------|------|------|
| 주문 CONFIRMED | **49** | 재고가 있던 만큼만 확정 |
| 주문 CANCELLED | **252** | 재고 소진 → `stock.rejected` → 보상·취소 |
| 재고 예약(RESERVED) | 49 | 확정분만. 취소분은 트랜잭션 롤백되어 예약 없음 |
| 포인트 적립 | 49건 | 확정 주문에만 적립 |

**초과 판매(overselling) 0건.** 300건 동시 유입에도 재고 49개면 정확히 49건만 확정되고,
나머지는 보상으로 취소됐다. 결제·적립도 확정 주문에만 발생해 데이터 정합성이 유지됐다.
재고 차감을 조건부 단일 UPDATE(`WHERE stock >= qty`)로 원자화한 덕분에 경쟁 상태가 없다.
